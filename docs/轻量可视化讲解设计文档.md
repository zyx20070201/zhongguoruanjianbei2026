# 轻量可视化讲解设计文档

## 1. 目标

这个设计面向一个新的 `visualize` 入口，但它不沿用当前已经混杂了 `visual_explainer`、`React-Chat`、`AI Studio V2` 上下文和多条渲染链路的旧实现。

它的目标很简单：

- 模仿人类教师讲课时的课件式讲解。
- 先生成正常 Markdown 讲稿。
- 再把讲稿切成多个 slide。
- 每个 slide 再生成一个按顺序展开的 timeline。
- 每个 slide 可以附带少量辅助图解。

一句话概括：

```text
prompt + 显式选中文件
  -> Markdown 讲稿
  -> slide 切分
  -> 每页 timeline
  -> 可选辅助图解
  -> 课件式可视化讲解
```

## 2. 设计原则

### 2.1 简单优先

字段越少越好，结构越稳越好。

这个功能不是为了生成一个复杂交互应用，而是为了生成一个像课堂 PPT 一样、能按层次展开的讲解内容。对大模型来说，简单结构反而更容易保持层次感，减少回答扁平化、字段乱填、结构漂移等问题。

### 2.2 新入口，不复用旧可视化讲解

建议在 `visualize` 下新增一个独立入口，例如：

- `light_visual_lesson`
- `slide_timeline_lesson`
- `teacher_slide_lesson`

具体命名后续可以再定，但原则是不复用旧的 `visual_explainer` 入口，也不走 React-Chat 自由代码生成那条路线。

### 2.3 上下文只接最小集

这个功能只使用两类输入：

- 用户 prompt
- 用户显式选择的文件内容

不接入以下复杂上下文：

- AI Studio V2 context capsule
- 自动检索
- 学习画像
- 长期记忆
- 推荐链路
- 多阶段 workflow trace

### 2.4 先讲清楚，再考虑动画

动画只是讲解顺序的可视化表达，不应该反过来主导内容生成。

因此：

1. 先有完整 Markdown 讲稿。
2. 再做 slide 切分。
3. 再对每页生成 timeline。
4. 最后再决定是否补一个辅助图解。

## 3. 用户心智模型

用户想要的不是“生成一个炫技的前端 demo”，而是：

- 一页一页讲
- 每页有明确标题
- 每页正文不是一次性全砸出来，而是逐条展开
- 必要时配一个简单图解帮助理解

更接近：

```text
老师讲课的 PPT + 页面内逐条展开
```

而不是：

```text
自由生成的交互沙盒 / 可执行网页应用 / 复杂动画引擎
```

## 4. 生成链路

建议采用严格串行设计。

### 4.1 第一步：生成 Markdown 讲稿

输入：

- user prompt
- selected files

输出：

- 一篇正常的 Markdown 讲稿

要求：

- 有层次
- 有小节
- 有自然段
- 讲解顺序清楚
- 不输出复杂 JSON

这一阶段的目标只是把“要讲什么”讲清楚。

### 4.2 第二步：按语义切分为 slides

切分器根据 Markdown 讲稿的语义结构切页。

切页依据可以是：

- 一级/二级标题
- 小节主题变化
- 段落长度
- 一个页面是否只讲一个核心点

目标不是机械按字数切，而是让每个 slide 对应一个自然的讲解单位。

每个 slide 最好只承担一种主要任务，例如：

- 引入概念
- 解释过程
- 展示例子
- 对比差异
- 总结规则

### 4.3 第三步：为每个 slide 生成 timeline

timeline 是页面内的讲解顺序，不是独立的大型状态机。

它的作用只是：

- 把正文拆成若干展开步骤
- 控制每一步显示什么
- 让讲解从“总述”变成“逐条展开”

每个 slide 的 timeline 应该很短，通常 3 到 6 步就够了。

### 4.4 第四步：为每个 slide 生成可选辅助图解

辅助图解不是强制项。

只有当某一页确实适合图解时才生成，例如：

- 流程
- 结构关系
- 表格执行过程
- 算法状态变化
- 概念对比

图解应当服务于 slide，而不是把 slide 变成一个复杂应用。

## 5. 最小数据结构

这里故意保持极简。

### 5.1 顶层结构

```ts
interface LightVisualLesson {
  title: string;
  slides: Slide[];
}
```

### 5.2 Slide 结构

你提的原则是每个 slide 只保留 `header` 和 `description` 作为核心字段，我同意这个方向。

建议最小保留为：

```ts
interface Slide {
  header: string;
  description: string;
  timeline?: TimelineStep[];
  visuals?: VisualBlock[];
}
```

其中：

- `header`：页标题
- `description`：这一页完整正文说明
- `timeline`：正文和图解的展开顺序
- `visuals`：这一页可选的辅助图解定义

### 5.3 Timeline 结构

timeline 不要复杂化成很多控制字段。

建议只保留：

```ts
interface TimelineStep {
  kind: 'text' | 'visual';
  content: string;
  visualIndex?: number;
}
```

含义如下：

- `kind: 'text'`：这一条是正文讲解步骤
- `kind: 'visual'`：这一条是图解出现或强调步骤
- `content`：这一条时间线要显示或讲述的内容
- `visualIndex`：如果该步骤对应某个图解，则引用 `visuals` 中的序号

这样可以让“正文 description 可以拆 timeline，图表等组件也可以拆 timeline”这件事统一在一个简单序列里完成。

### 5.4 Visual 结构

图解也保持极简，不做过度建模。

```ts
interface VisualBlock {
  type: 'diagram' | 'table' | 'formula' | 'code' | 'image_hint';
  content: string;
}
```

说明：

- `diagram`：流程图、结构图、关系图等
- `table`：表格或对比表
- `formula`：公式
- `code`：代码片段
- `image_hint`：给后续图像或示意图生成器的简短说明

这里不引入一堆坐标、颜色、动画动作、对象列表、targetIds、renderer block 等复杂字段。

## 6. 推荐输出示例

```json
{
  "title": "冒泡排序可视化讲解",
  "slides": [
    {
      "header": "冒泡排序的核心思想",
      "description": "冒泡排序通过重复比较相邻元素，把较大的元素逐步交换到序列右侧。每一轮结束后，当前未排序部分中的最大值都会被放到正确位置。",
      "timeline": [
        {
          "kind": "text",
          "content": "先理解目标：每一轮都把一个最大元素送到右边。"
        },
        {
          "kind": "text",
          "content": "排序时只比较相邻元素，如果顺序不对就交换。"
        },
        {
          "kind": "visual",
          "content": "展示一轮扫描后最大值移动到最右侧。",
          "visualIndex": 0
        }
      ],
      "visuals": [
        {
          "type": "diagram",
          "content": "数组从左到右比较，相邻逆序则交换，最大元素逐步冒到最右端。"
        }
      ]
    },
    {
      "header": "第一轮扫描发生了什么",
      "description": "从第一个元素开始，依次比较相邻位置。较大值不断向右移动，因此第一轮完成后，最后一个位置就是当前最大值。",
      "timeline": [
        {
          "kind": "text",
          "content": "比较第 1 对相邻元素。"
        },
        {
          "kind": "text",
          "content": "如果左边更大，就交换位置。"
        },
        {
          "kind": "text",
          "content": "继续向右扫描，直到本轮结束。"
        }
      ]
    }
  ]
}
```

## 7. 为什么这个结构够用

这个结构虽然简单，但已经能表达四件关键事情：

1. 一共有多少页
2. 每一页讲什么
3. 每一页按什么顺序展开
4. 某一页是否需要图解辅助

对当前目标来说，这已经足够支撑一个课件式讲解体验。

反过来说，下面这些复杂能力暂时都不是必须的：

- 每个元素精确坐标
- 多种动画 easing
- 图元级对象树
- 多 renderer 并存
- 大量状态 patch
- 自动路由到不同前端沙盒

这些字段一多，就会增加模型干扰，反而更容易让结果变乱。

## 8. 生成约束建议

为了避免大模型输出扁平化，建议在生成时明确约束：

- `header` 必须短，像 PPT 页标题
- `description` 必须是完整讲解，而不是关键词堆砌
- `timeline` 是从 `description` 中拆出来的逐步讲解，不是重复粘贴原文
- 每个 slide 最好多讲一个核心点，不要一页塞太多主题
- `visuals` 只有在确实需要时才出现
- 没有必要时可以不生成 `visuals`
- `timeline` 数量控制在少量步骤，避免页面内过碎

## 9. 与旧实现的边界

这个新入口建议与现有几套机制明确隔离：

### 9.1 不复用旧 visual explainer 结构

不沿用以下复杂内容：

- section objects
- renderer blocks
- targetIds
- action graph
- staged visual pipeline

### 9.2 不走 React-Chat 自由可视化协议

不依赖：

- `~~~REACT_VIZ`
- `~~~HTML_VIZ`
- 自由代码块解析
- 交互沙盒优先

因为这个功能的目标不是“生成一个应用”，而是“生成一套课件式讲解页面”。

### 9.3 不接 AI Studio V2 全量上下文

只允许：

- user prompt
- 用户显式选择的文件

这样有两个好处：

- 可控
- 结果来源清晰

## 10. 后续实现建议

虽然这里不写代码，但实现时建议仍然分成几个非常简单的阶段：

1. Markdown Draft  
   只生成讲稿。

2. Slide Split  
   把讲稿切成 slide。

3. Timeline Expansion  
   把每页正文拆成少量 timeline 步骤。

4. Optional Visual Hint  
   只在必要时给某页补一个极简图解。

5. Frontend Playback  
   前端只负责：
   - 翻页
   - 页面内按 timeline 逐步展开
   - 渲染少量图解

## 11. 结论

这个“轻量可视化讲解”入口应该被定义成一个极简、稳定、可控的课件生成器。

它不是旧 `visual_explainer` 的小修小补，也不是 React-Chat 沙盒的另一层包装，而是一条新的轻量链路。

它的核心不是炫技，而是教学表达：

- 用 Markdown 保证内容完整
- 用 slide 保证层次分明
- 用 timeline 保证逐步讲解
- 用少量图解保证理解辅助

如果后续需要扩展，也应该在这个极简结构上渐进增加，而不是一开始就塞进太多字段。
