# 02 可行性分析报告

## 一、项目需求分析概况

本项目面向大学生学习全流程，核心需求是把课程空间、学习现场、资料库、知识图谱、AI 对话、学习计划、学习画像和资源生成组织成一个连贯系统。数据库与业务层需要支持用户隔离、资料复用、过程记录、智能问答和结果追踪。

从使用场景看，学生在一门课程中往往会同时面对课件、论文、实验任务、课堂笔记、复习计划和项目实践材料。如果这些资料分散在聊天记录、网盘、浏览器收藏和本地文件夹中，学习过程就很难被持续记录，也很难形成可复用的知识结构。因此，本项目不仅要实现单点功能，还要把学习任务、资源组织、智能辅助和阶段反馈放在同一个工作空间中，使用户能够围绕课程目标持续推进。

从管理角度看，该项目需求具有明显的软件工程特征：既包含前端交互、后端接口和数据库模型，也包含大模型工具调用、资料检索、知识图谱生成和多模态分析等智能能力。需求之间存在前后依赖，例如资料库是知识图谱和 AI 问答的基础，用户画像又依赖学习过程记录。因此，可行性分析需要同时考虑技术成熟度、团队组织方式、成本边界、部署环境和课程提交节点。

本项目的需求不是抽象地“做一个 AI 学习平台”，而是围绕一条具体用户路径展开：登录系统、创建 Workspace、添加资料、在 AI Terminal 中提出目标、生成或进入 Workbench、在多面板工作区完成学习、把生成物保存回资料库，并通过知识图谱、Planning、Profile 和 Intelligence 标签页查看学习状态。只要这条路径能稳定运行，项目就具备基本可用性；AI Studio、多模态分析、闪卡和 Agent V2 则在此基础上增强系统表现力。

## 二、可行性要素分析

### 1. 技术可行性

项目技术栈已经明确，前端、后端和数据库都采用成熟方案，且项目中已有相应工程代码和设计文档。

- 前端：React + TypeScript + Vite。
- 后端：Node.js + Express。
- 数据层：Prisma + SQLite，本地开发简单，迁移成本低。
- 智能能力：现有设计已覆盖检索、图谱、规划、Agent 和多模态处理。

因此，从技术栈成熟度和可实现性看，项目可行。

React、TypeScript 和 Vite 适合快速构建复杂交互界面，能够支撑课程空间、工作台、资料列表、图谱视图和 AI 对话等多种页面形态。Node.js 与 Express 便于搭建轻量后端服务，Prisma 能统一管理数据模型和迁移过程，降低数据库结构变动带来的维护成本。对于智能能力部分，项目采用大模型驱动的 Codex 类开发与辅助实现方式，可以提升原型搭建、代码修改和文档整理效率，但仍需要通过测试和评审保证生成内容与项目需求一致。

从已有实现看，系统并不是只停留在概念阶段。前端已经围绕 Workspace Shell、WorkbenchPage、LearningTerminal、AddSourcesDialog、AIStudioPanel 等核心界面组织用户操作；后端已经围绕 workspace、workbench、file、learning、studio 等路由和服务组织业务能力；数据库模型中也包含 Workspace、Workbench、FileSystemObject、LearningGoal、LearningPlan、LearningEvent、KnowledgeChunk、CourseKnowledgeConcept、SavedMemory、StudioArtifact、Flashcard 等对象。这些实现基础说明项目技术可行性较强。

### 2. 经济可行性

项目以课程项目和竞赛作品为主，成本主要来自开发时间、大模型辅助开发工具、少量算力和部署服务，不需要采购专用硬件或建设复杂机房环境。实际投入中，主要金钱成本集中在购买大模型驱动的 Codex 类产品、模型调用额度，以及产品在服务机或云服务器上的部署费用。按照课程项目和演示原型的规模估计，这部分现金支出可以控制在 1500 元以内。

从收益角度看，项目并不以短期商业变现为主要目标，而是以课程交付、竞赛展示和能力沉淀为核心价值。项目可以复用到软件杯答辩、课程实践、学习工具原型和后续论文/项目申报材料中，因此投入产出比是可接受的。若将团队成员的学习时间和开发时间也折算为成本，则应在成本计划中单独列出课程估算口径，避免与真实现金支出混淆。

### 3. 进度可行性

当前项目已经完成了基础架构、功能说明和部分高级设计，说明主体工作不是从零开始。只要保持阶段交付和范围控制，在课程节点前完成报告和演示是可行的。

项目进度采用分阶段推进方式，先完成需求梳理、范围裁剪和基础架构，再实现课程空间、学习现场、资料库等主流程，最后接入知识图谱、学习规划、AI Terminal、多模态分析等增强能力。由于课程报告和软件杯材料可以共享同一套项目背景、模块划分和时间线，文档工作不会完全从零开始，进度风险主要来自后期需求扩张和演示环境不稳定。

### 4. 组织可行性

本项目实际采用 3 人组织方式推进。团队中一人负责全栈开发，覆盖前端页面、后端接口、数据库模型和主要工程实现；一人负责运营维护、测试、演示环境和部署检查；一人负责需求分析、开源项目调研、竞品与论文资料查找，并提供高层次技术路线和论文方向指导。

这种组织方式人数较少，但沟通链路短，决策效率高，适合课程项目和竞赛原型快速迭代。为了避免职责过度集中，项目需要明确每个人的交付物：全栈开发人员保证核心功能可运行，运维测试人员保证环境稳定和问题可复现，需求与研究人员保证功能方向与赛题、课程要求一致。只要保持任务清单和版本记录，3 人团队能够支撑当前项目范围。

### 5. 运维可行性

系统可在本地开发环境和轻量云服务器环境中运行。项目早期使用 SQLite 便于快速开发和本地演示，后续已经迁移到阿里云数据库，以提升远程访问、数据持久化和部署稳定性。阿里云数据库能够更好地支撑多人访问和正式演示，也便于后续进行备份、权限配置和环境隔离。

运维方面，本项目需要关注端口配置、环境变量、数据库连接、模型服务额度和演示数据准备。由于课程答辩和软件杯展示都强调可演示性，因此应保留本地最小运行方案和云端部署方案两套路径。一旦云数据库或外部模型接口出现异常，可以通过备用数据、静态截图或降级演示保证交付。

### 6. 安全与合规可行性

系统涉及用户数据、学习资料和对话记录，必须保证访问隔离、会话安全和内容可追溯。现有架构已经具备认证、权限与数据归属分层，合规风险可控。

在课程项目阶段，安全目标主要是防止跨用户数据访问、避免敏感配置暴露、保证演示账号与真实账号隔离。对于用户上传资料和 AI 对话内容，应在报告和演示中避免使用真实隐私数据，尽量使用课程样例、公开资料或脱敏内容。后续若产品化，还需要进一步补充日志审计、权限分级、数据备份和内容安全策略。

## 三、项目设计方案

建议采用“课程空间 + 学习现场 + 资料资源 + 智能能力”的四层组织方式。

| 层次 | 作用 |
| --- | --- |
| 课程空间 | 作为整门课程或主题的总容器 |
| 学习现场 | 面向某个任务的具体工作区 |
| 资料资源 | 文件、网页、笔记、生成物和引用来源 |
| 智能能力 | 检索、图谱、规划、对话、生成、分析 |

这种方案能把业务边界、数据边界和管理边界统一起来，便于课程报告和后续开发同步推进。

四层结构的优势在于既便于用户理解，也便于开发和管理。课程空间对应学习主题的长期容器，学习现场对应一次具体任务或学习活动，资料资源提供检索和生成的依据，智能能力则围绕资料与任务提供问答、总结、规划和分析。各层之间既有依赖，又能相对独立迭代，适合用作软件项目管理中的工作分解结构。

在实际实现中，项目应优先保障“用户进入课程空间、创建学习现场、导入资料、调用智能能力、保存学习结果”这一条核心闭环。对于多模态分析、复杂 Agent 编排和高级可视化，可以作为增强功能逐步完善，不应影响主流程交付。

因此，系统设计方案可以进一步落到以下业务映射：Workspace Shell 负责课程级导航和 Workbenches、Knowledge、Knowledge Graph、Planning、Profile、Intelligence、AI Terminal 等标签页；Workbench 负责具体学习任务中的资料预览、笔记、代码、AI 助手和 AI Studio 多面板协作；FileSystemObject 负责统一保存上传资料、网页来源、笔记和 AI 生成产物；知识索引和向量检索负责把资料变成可引用上下文；学习画像和事件记录负责让规划与反馈具备个性化依据。

## 四、人员配置和培训计划

### 人员配置

- 全栈开发负责人：负责前端页面、后端接口、数据库模型、核心业务逻辑和主要功能联调。
- 运维测试负责人：负责部署环境、数据库连接、测试验证、问题复现、演示数据和运行稳定性检查。
- 需求与研究负责人：负责需求分析、开源项目调研、论文与技术路线参考、赛题理解和课程报告内容校对。
- 项目统筹由三人共同承担，其中范围控制、节点检查和文档一致性需要在每个阶段进行同步确认。

### 培训计划

- Prisma 数据建模与迁移。
- 课程空间与学习现场的业务边界。
- 软件项目管理各文档之间的关联。
- Agent、检索和图谱的基础接口使用。
- Codex 类 AI 辅助开发工具的使用规范，包括任务拆分、代码审查、生成内容验证和文档润色。
- 阿里云数据库连接、备份、权限配置和演示环境排查方法。

培训方式以项目内分享和边做边学为主。由于团队规模较小，不适合安排过重的集中培训，而应把培训内容嵌入真实任务中，例如在数据库迁移时同步讲解 Prisma，在演示部署时同步讲解云数据库连接，在报告整理时同步讲解项目管理文档之间的对应关系。

## 五、项目主要风险

1. 需求范围不断扩大，导致报告与实现口径不一致。
2. 智能能力较多，容易出现依赖外部服务不稳定的问题。
3. 多模块并行开发时，数据结构和接口容易出现耦合。
4. 临近课程节点时，文档与演示准备时间不足。
5. 3 人团队中全栈开发任务较重，若关键人员时间受限，会影响整体进度。
6. 云数据库、模型服务和部署环境存在配置差异，可能导致本地可运行但线上演示异常。
7. Codex 类工具可以提高效率，但生成内容仍需要人工审查，不能直接替代测试和设计判断。

针对上述风险，项目应提前设定范围基线，明确必须交付的核心功能和可延期的扩展功能。智能能力要准备降级路径，数据库和部署环境要保留连接配置记录，课程报告应以当前真实实现为依据滚动更新。

## 六、结论和建议

本项目在技术、经济、组织和进度上均具备实施条件，结论是可行。建议采取“核心功能优先、智能能力分层、文档同步推进”的策略，把软件项目管理作业与当前软件杯项目统一起来，避免两套材料相互脱节。

综合分析可知，项目的最大优势是题目真实、功能边界清楚、技术栈成熟，并且已经具备一定工程基础；主要限制是团队规模较小、智能能力依赖外部服务、后期文档和演示工作量较大。因此，实施过程中不应盲目追求功能数量，而应围绕课程空间、学习现场、资料库和 AI 辅助学习主流程建立稳定闭环。

建议在课程提交前完成三项收束工作：第一，确认所有报告中的项目名称、团队规模、成本口径、时间节点和模块划分一致；第二，准备本地与云端两套演示环境，并对阿里云数据库连接进行检查；第三，对 Codex 辅助生成或修改的代码和文档进行人工复核，确保内容准确、可解释、可答辩。

## 七、其他重要意见

课程报告不宜追求空泛的大而全，而应以当前真实项目为依据，优先保证：

- 项目范围清晰。
- 模块关系明确。
- 每份报告内容和项目状态一致。
- 结论可支撑后续计划、成本、风险和质量文档。
