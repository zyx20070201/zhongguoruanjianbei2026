# 09 质量管理报告

## 一、质量管理概述

本报告记录项目在开发与整理过程中对质量的实际管理思路，重点包括质量计划落实、技术评审、测试、缺陷跟踪、过程检查和改进。

质量管理与质量计划不同，质量计划强调“准备怎样保证质量”，质量管理则强调“实际怎样执行质量控制”。本项目在推进过程中，需要同时管理软件功能质量、数据质量、演示质量和文档质量。由于项目使用 Codex 类 AI 辅助开发工具，并涉及阿里云数据库和外部智能服务，质量管理还需要关注生成内容复核、部署环境检查和降级方案验证。

本项目的质量管理目标可以概括为：核心流程可运行，关键数据不混乱，智能能力有说明和降级路径，课程报告与真实项目一致，最终演示能够稳定完成。

## 二、质量管理工作顺序

1. 制订项目质量计划。
2. 开发过程中进行技术评审。
3. 开发过程中进行软件测试。
4. 对缺陷进行跟踪和修复。
5. 对整个过程进行检查。
6. 持续改进文档和实现。

实际执行时，质量管理并不是等到开发结束后才开始。需求阶段需要检查项目范围是否过大，设计阶段需要检查模块边界和数据模型是否合理，开发阶段需要检查接口和页面是否联通，测试阶段需要检查主流程和异常情况，收尾阶段需要检查文档、截图、PPT 和演示脚本是否一致。每一阶段的质量活动都为下一阶段降低返工风险。

## 三、组织保证

当前项目的质量责任可分为四层：

- 管理层：把控范围、节点和最终提交。
- 开发层：保证功能正确与可维护。
- 测试层：检查主流程、异常和回归。
- 文档层：保证报告、说明和实现一致。

结合 3 人团队实际，质量保证主要依靠角色互补。全栈开发负责人对代码、接口和数据库修改进行自检；运维测试负责人通过启动、部署、回归和演示流程发现问题；需求与研究负责人检查功能方向、论文参考和课程报告表述是否符合项目定位。对于重要内容，尽量由非原作者进行复核，减少自查遗漏。

## 四、阶段性质量控制

### 1. 需求与设计阶段

重点检查项目是否真正围绕课程学习场景展开，是否把 Workspace、Workbench、资料、图谱、规划和画像连接成一个整体。

本阶段的质量管理重点是防止需求漂移。团队需要确认哪些功能属于核心闭环，哪些属于展示增强，哪些只作为后续规划。需求和设计文档必须与软件杯赛题、课程报告要求和现有工程基础对应起来，避免文档描述超出实际实现能力太多。

### 2. 开发阶段

重点检查：

- 接口是否与数据模型匹配。
- 页面是否符合实际使用流程。
- 核心路径是否能稳定走通。
- Codex 辅助生成的代码是否经过人工审查和运行验证。
- 数据库结构变更是否同步到本地环境和阿里云数据库环境。

开发阶段的质量管理主要通过自测、联调和局部评审完成。每完成一个模块，应至少检查页面是否能打开、接口是否返回预期数据、异常情况是否有反馈、相关报告是否需要同步修改。对于智能能力模块，还要检查外部服务不可用时是否存在降级说明。

在实际业务中，联调质量尤其重要。例如 Workspace 删除或复制会影响 Workbench、FileSystemObject、LearningPlan、KnowledgeChunk 和 StudioArtifact 等关联数据；资料删除会影响 Workbench 资源列表、已打开面板和知识索引；AI Studio 生成物需要既能在当前 Workbench 打开，又能保存到资料库；AI Terminal 创建 Workbench 或计划时需要经过用户确认，并能在对应标签页中看到结果。这些跨模块联动是质量管理的重点。

### 3. 测试阶段

重点检查：

- 登录与权限。
- 课程空间和学习现场的 CRUD。
- 资料管理与检索。
- 智能能力的降级和异常处理。
- 阿里云数据库连接和部署环境启动。
- 答辩演示路径、截图和备用材料。

测试阶段不只验证功能是否“点得动”，还要验证是否适合演示和提交。测试人员应按照答辩脚本走一遍完整流程，记录卡顿、报错、页面空状态和数据不一致问题。对于无法及时修复的问题，应判断是否影响核心流程，并决定修复、降级或在报告中说明。

质量管理记录应尽量围绕真实场景描述。例如“上传数据库课程 PDF 后建立知识索引，AI Terminal 能根据资料回答 JOIN 与 GROUP BY 的问题”“在 Workbench 中打开笔记和代码面板后刷新页面，布局和资源状态能够恢复”“AI Studio 生成的测验保存到 Generated 文件夹并能再次打开”。这种记录比“完成测试”更能体现项目质量。

## 五、缺陷跟踪与过程改进

项目质量管理不是只找 bug，更要把缺陷归因到流程问题上。

- 如果功能遗漏，说明需求拆分不够细。
- 如果文档不一致，说明版本控制不够严格。
- 如果演示不稳定，说明回归测试太晚。

因此，缺陷处理后还要同步修正文档、检查清单和任务划分。

缺陷跟踪建议按严重级别处理。严重缺陷包括系统无法启动、登录失败、核心数据丢失、跨用户数据访问和报告关键数据错误；一般缺陷包括局部页面状态异常、个别接口返回不稳定和文档表述不够准确；轻微缺陷包括样式细节、措辞优化和非核心增强功能体验问题。缺陷关闭前，应确认修复是否引入新的问题。

## 六、软件评审内容

质量管理应覆盖：

- 需求定义
- 设计
- 代码
- 测试计划
- 测试用例

本项目中，课程报告自身也应作为一种“管理对象”参与评审，因为它直接反映项目状态。

评审时应重点查看五类一致性：需求与功能是否一致，设计与代码是否一致，测试与风险是否一致，成本与资源是否一致，报告之间是否一致。例如，可行性分析中写到 3 人团队、Codex 类工具和阿里云数据库，资源计划、成本计划和总结报告也应保持同样口径。

## 七、质量度量

建议使用以下度量：

- 缺陷密度
- 缺陷清除率
- 回归通过率
- 关键流程可用率
- 文档一致性

这些指标不必追求复杂统计，但要能反映项目是否在变好。

结合课程项目特点，还可以增加三个轻量指标：核心演示链路通过率、报告一致性问题数量、外部服务降级方案覆盖率。核心演示链路通过率用于衡量答辩稳定性；报告一致性问题数量用于衡量文档质量；降级方案覆盖率用于衡量智能服务和部署环境风险是否可控。

## 九、度量结果分析

从阶段性质量看，本项目目前的强项是功能范围完整、文档基础充分；弱项是自动化测试和最终演示回归仍需强化。质量管理后续应继续围绕“缺陷是否关闭、演示是否稳定、报告是否一致”三个指标推进。

自动化测试不足并不意味着质量无法控制，但需要用更严格的手工回归和清单检查弥补。尤其在课程提交前，应把登录、课程空间、学习现场、资料库、知识图谱、AI Terminal 和学习规划作为固定检查项。对于 AI Studio、多模态分析等高级功能，应根据稳定性决定是否放入主演示路径。

文档质量方面，目前 11 份报告已经形成统一框架，后续重点是补充项目真实信息，包括 3 人团队分工、Codex 辅助开发、阿里云数据库迁移和实际现金成本控制。这些内容能够使报告更可信，也能减少答辩时解释成本。

更进一步，质量管理报告应体现业务质量结论：系统的核心价值不在某一个 AI 回答，而在 Workspace、资料库、Workbench、检索、规划和生成物之间能否形成闭环。如果用户上传资料后无法被检索，或生成内容无法保存回学习现场，智能能力就只是临时演示；只有这些产物能沉淀为课程资源，系统才真正符合“AI Learning Workspace”的定位。

## 十、当前阶段的质量结论

当前项目已具备较完整的结构和较清晰的功能边界，质量管理重点已从“搭框架”转向“查缺陷、保一致、稳演示”。后续最需要补强的是自动化回归、文档同步和最终演示路径。

质量结论是：项目可以进入收尾和集中校对阶段，但不适合继续无节制增加新功能。后续新增内容必须满足三个条件：不破坏核心流程、不显著增加部署和测试压力、能够在报告中同步说明。否则应作为后续优化项记录，而不是加入当前交付范围。

## 十一、经验总结

1. 质量管理要早，不要等最后一周。
2. 文档和代码要一起管。
3. 复杂功能要先留降级方案。
4. 对课程作业来说，稳定可交付比炫技更重要。
5. AI 辅助开发提高效率，但必须保留人工复核。
6. 云端部署和数据库配置要提前检查，不能只在答辩前临时处理。
7. 报告之间的数据口径越早统一，后期返工越少。

总体来看，本项目的质量管理体现了课程项目的实际特点：资源有限、时间固定、功能较多，因此必须把稳定性和一致性放在首位。只要持续执行缺陷跟踪、文档校对和演示回归，项目质量可以满足课程提交和软件杯展示的基本要求。
