# 07 质量计划

## 一、质量计划目标

本项目的质量目标是：功能可用、数据一致、界面清晰、文档统一、演示稳定。

具体来说，功能可用是指用户能够完成从课程空间到学习现场、资料管理和智能辅助的核心流程；数据一致是指用户、课程、资料、对话、计划和画像等对象之间的关系清楚，不能出现跨用户数据混乱；界面清晰是指主要操作路径容易理解，页面状态和反馈明确；文档统一是指 11 份课程报告使用同一项目名称、同一团队分工、同一时间线和同一成本口径；演示稳定是指本地环境和云端部署至少有一条路径可以顺利展示。

## 二、质量目标分解

| 维度 | 质量目标 |
| --- | --- |
| 功能质量 | 核心流程完整，可闭环运行 |
| 数据质量 | 数据模型清晰，主外键和归属关系正确 |
| 代码质量 | 命名统一，结构清楚，便于维护 |
| 文档质量 | 11 份课程报告口径一致 |
| 演示质量 | 可稳定启动，关键流程可演示 |
| 安全质量 | 用户隔离和访问控制明确 |

质量目标还应结合课程项目特点进行解释。由于本项目不是正式商业系统，质量管理不追求复杂指标堆叠，而是优先保证最终提交可信、可运行、可解释。对课程评分和答辩来说，文档与实现一致、演示路径稳定、关键风险有说明，往往比单个扩展功能更重要。

质量目标还需要映射到真实功能。账号与安全模块的质量重点是登录、会话 Cookie、受保护路由和用户数据隔离；Workspace 的质量重点是课程创建、切换、复制、删除和标签页导航；Workbench 的质量重点是资源绑定、多面板布局、状态保存和再次打开恢复；资料库的质量重点是上传、网页导入、文本粘贴、预览、删除和索引；AI Terminal 的质量重点是上下文选择、流式输出、历史会话和审批；AI Studio 的质量重点是生成物可保存、可打开、可用于学习复盘。

## 三、参考标准和规范

- 软件项目管理课程报告内容依据。
- 当前项目的系统功能说明和需求分析。
- 前后端代码命名、目录和接口约定。
- 数据库字段、表关系和数据归属规范。
- Codex 类 AI 辅助开发内容的复核规范。
- 阿里云数据库和部署环境的配置检查规范。

参考标准主要来自三类材料。第一类是课程要求，决定报告结构和管理文档之间的关系；第二类是项目自身文档，决定功能范围和术语；第三类是工程实践规范，决定代码、接口、数据库和部署是否可维护。所有质量检查都应回到这三类依据上，避免凭主观感觉判断质量。

## 四、组织保证机制

- 项目负责人负责质量底线和版本冻结。
- 开发人员对自己模块负责自测与修复。
- 测试与文档人员负责检查一致性和遗漏项。
- 所有修改必须同步到文档或变更说明。

结合本项目 3 人团队实际，质量责任可进一步拆分为：全栈开发负责人对功能正确性、接口一致性和代码可维护性负责；运维测试负责人对环境稳定性、回归测试、演示数据和部署配置负责；需求与研究负责人对需求完整性、论文资料参考、课程报告口径和功能优先级负责。三类责任共同构成项目质量保障。

## 五、质量任务

| 质量任务 | 具体内容 |
| --- | --- |
| 流程说明 | 明确从需求、设计、开发、测试到收尾的工作流 |
| 流程实施指导 | 对接口、数据模型、文档命名和提交材料进行约束 |
| 关键成果评审 | 评审需求、设计、测试计划、报告和演示稿 |
| 缺陷跟踪 | 建立问题记录，明确严重级别、责任人和关闭状态 |
| 过程改进 | 根据缺陷和延期原因调整计划 |

质量任务应贯穿项目全生命周期。需求阶段重点防止范围不清，设计阶段重点防止数据关系混乱，开发阶段重点防止接口和页面脱节，测试阶段重点防止主流程中断，收尾阶段重点防止文档不一致和演示材料遗漏。每一类质量任务都应有明确产物，例如评审记录、测试清单、缺陷列表或报告修改记录。

## 六、质量控制策略

1. 需求评审：确认是否符合课程要求和项目范围。
2. 设计评审：检查数据模型、模块边界和接口。
3. 代码评审：检查命名、重复逻辑、异常处理和权限。
4. 测试评审：检查主流程、异常流和回归情况。
5. 文档评审：检查内容、序号、命名和术语一致性。

除上述策略外，还需要对 AI 辅助生成内容进行专门控制。Codex 类工具生成的代码应至少经过语义检查、运行验证和必要的人工修改；生成或扩写的文档应核对项目事实、团队人数、成本数字和时间节点，不能直接把提示性文字保留在最终报告中。对于数据库和部署相关修改，还应检查本地环境与阿里云数据库环境是否一致。

质量控制应采用场景化测试。建议以“数据库课程复习”为样例，完整测试：创建 Workspace、上传 SQL 资料、创建 JOIN 复习 Workbench、在资料面板查看文件、在笔记面板记录总结、使用 AI 助手解释选中文本、通过 AI Terminal 生成复习计划、查看知识图谱概念、生成闪卡或测验并保存。这个场景覆盖大多数核心模块，比孤立点击页面更能反映系统质量。

## 七、质量属性优先级

优先级从高到低建议为：

1. 正确性
2. 稳定性
3. 数据一致性
4. 可演示性
5. 性能
6. 扩展性

优先级排序体现了课程项目的现实约束。正确性和稳定性排在最前，是因为系统首先要能按预期运行；数据一致性关系到资料、图谱和学习画像等模块是否可信；可演示性直接影响答辩效果；性能和扩展性也重要，但在课程原型阶段可以通过分页、懒加载、降级方案和后续规划逐步完善。

## 八、关键文档要求

关键文档包括：

- 程序员说明或开发说明。
- 测试计划。
- 配置管理计划。
- 11 份课程报告。
- 答辩 PPT 和演示脚本。

这些文档必须使用同一项目名称、同一模块划分、同一时间线和同一成本/进度口径。

文档质量要求不仅包括格式整齐，还包括内容之间相互支撑。例如，可行性分析中的组织可行性应与资源计划中的 3 人分工一致，成本计划中的现金支出应与项目建议书中的投资估算一致，风险计划中的外部服务风险应在风险管理报告中继续跟踪，质量计划中的检查标准应在质量管理报告中体现执行结果。

对于本项目，文档还应尽量使用系统内真实概念，而不是泛称“模块一、模块二”。例如应明确写出 Workspace 是课程级容器、Workbench 是任务级学习现场、FileSystemObject 是资料统一模型、KnowledgeChunk 和 CourseKnowledgeConcept 支撑检索与图谱、LearningPlan 和 LearningEvent 支撑规划与画像、StudioArtifact 和 Flashcard 支撑资源生成与复习。这样报告才能体现项目业务设计。

## 九、潜在质量问题与措施

- 功能遗漏：用需求清单逐项核对。
- 页面错位：统一组件与布局规范。
- 数据关系混乱：先修正模型，再做功能。
- 文档口径分裂：统一项目术语和时间线。
- 演示崩溃：准备降级版演示路径。
- AI 生成内容失真：人工复核事实和数据。
- 云数据库配置错误：保留本地备用环境和连接检查表。
- 团队资源不足：冻结低优先级增强项。

对于潜在质量问题，处理时应区分严重级别。影响系统启动、登录、核心数据和课程提交格式的问题属于严重问题，必须优先修复；影响局部展示效果或增强功能体验的问题可以在不影响主流程的情况下延后；仅影响样式美观或扩展能力的问题可作为后续优化项记录。

## 十、评审和检查要求

质量计划应覆盖：

- 需求说明评审。
- 设计文档评审。
- 测试计划评审。
- 功能性审核。
- 物理性审核。
- 管理评审。

评审标准包括：是否符合需求、是否可运行、是否有明确责任人、是否与课程报告一致、是否影响最终提交。

评审可以采用轻量方式进行，但必须有结果。每次评审后应明确通过、需修改或暂缓，并记录主要问题。对于课程报告，评审重点是标题编号是否完整、项目事实是否准确、表格数字是否前后一致、是否仍保留括号提示或草稿语气。

## 十一、配置管理与问题处理

- 使用版本控制保存代码和文档。
- 需求变更必须有记录。
- 问题统一登记、分派、跟踪和关闭。
- 关键产物如数据库结构、接口和报告必须有版本基线。

配置管理范围应包括代码、数据库迁移、环境变量说明、部署配置、课程报告、演示截图和 PPT。特别是阿里云数据库连接信息和模型服务配置，不能散落在个人聊天或临时记录中，应以安全方式记录必要说明。对于文档变更，也要避免不同报告引用不同版本的项目范围。

## 十二、问题报告和处理系统

问题记录建议包含：问题编号、发现时间、所属模块、严重级别、复现步骤、负责人、处理状态和关闭说明。严重问题优先处理影响系统启动、数据安全、核心流程和提交格式的问题。

问题处理流程建议为：发现问题后先记录复现条件，再判断严重级别和责任人；修复后由非修复人员尽量进行复查；如果问题影响报告内容，则同步修改相关报告；如果问题来自流程缺陷，例如需求遗漏或测试太晚，则在阶段复盘中记录改进措施。这样问题报告系统就不只是缺陷清单，而能推动过程改进。

## 十三、采用的工具、技术和方法

- 代码检查：TypeScript 检查、静态审阅。
- 接口检查：手工冒烟测试和接口联调。
- 文档检查：目录、术语、序号和内容依据对照。
- 演示检查：按答辩流程跑通关键页面。
- 环境检查：本地环境、服务机、阿里云数据库连接和模型服务状态。
- AI 内容检查：对 Codex 辅助生成的代码和文档进行人工确认。

工具和方法的选择应服务于项目规模。对于课程项目，不必引入过重的平台，但要保证检查动作真实发生。手工冒烟测试、清单式文档校对、阶段性截图留存和关键命令记录，都是成本较低但效果明确的质量管理方法。

## 十四、补充说明

质量计划的重点不是写得最漂亮，而是能把“怎么保证质量”写清楚。对于本项目，质量首先体现在课程报告与实际项目保持一致，其次才是界面、性能和扩展能力。

本项目质量计划还要体现智能开发工具带来的新要求。Codex 类工具可以提高效率，但不能替代团队对需求、代码和文档的判断。最终交付物必须由团队成员确认，确保系统能够运行、报告能够解释、答辩能够回答。只有把工具效率和人工质量控制结合起来，项目质量才是可靠的。
