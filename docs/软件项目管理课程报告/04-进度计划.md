# 04 进度计划

## 一、进度计划说明

本项目的进度计划以 2026 年 3 月 1 日到 2026 年 6 月 17 日为课程交付窗口，按“先立项、再核心功能、再智能能力、最后收尾”的顺序推进。

进度计划同时服务于两个目标：一是保证软件杯项目原型能够形成可演示的主流程，二是保证软件项目管理课程要求的 11 份报告能够按统一口径完成。由于两类交付物相互关联，进度安排不能只关注代码开发，还必须把需求文档、测试记录、截图材料、答辩脚本和报告修订纳入同一计划。

## 二、工作执行顺序

1. 需求梳理与范围确认。
2. 数据模型和基础架构搭建。
3. 课程空间、学习现场、资料库等核心业务实现。
4. AI Terminal、知识图谱、学习规划、学习画像等智能能力接入。
5. 测试、修复、文档和演示材料整理。

在执行过程中，每一阶段都需要形成可检查的中间产物。需求阶段输出范围清单和模块边界，架构阶段输出数据模型和接口草案，核心业务阶段输出可运行主流程，智能能力阶段输出可演示功能点，收尾阶段输出报告、截图、PPT 和演示路径。只有产物可检查，进度状态才不只是主观描述。

本项目的进度检查应围绕真实用户流程设置，而不是只看模块名称是否写完。第一条检查线是“登录 -> Workspace Shell -> 添加资料 -> AI Terminal -> Workbench -> 保存生成物”；第二条检查线是“资料库 -> 知识索引 -> 检索问答 -> 知识图谱/学习规划”；第三条检查线是“Workbench 多面板 -> AI Studio -> 测验/闪卡/可视化资源 -> 回写资料库”。这些链路能跑通，进度才具有产品意义。

## 三、阶段计划

| 阶段 | 任务 | 起止时间 | 依赖关系 |
| --- | --- | --- | --- |
| 立项阶段 | 需求分析、范围裁剪、WBS | 2026-03-01 至 2026-03-10 | 无 |
| 基础架构阶段 | 前后端骨架、认证、数据库模型 | 2026-03-11 至 2026-03-25 | 立项阶段 |
| 核心业务阶段 | Workspace、Workbench、资料库 | 2026-03-26 至 2026-04-15 | 基础架构阶段 |
| 智能能力阶段 | AI Terminal、检索、知识图谱、规划 | 2026-04-16 至 2026-05-20 | 核心业务阶段 |
| 扩展完善阶段 | 学习画像、Studio、多模态分析、Agent | 2026-05-21 至 2026-06-10 | 智能能力阶段 |
| 收尾阶段 | 测试、文档、截图、答辩稿 | 2026-06-11 至 2026-06-17 | 全部阶段 |

阶段计划中的时间并非平均分配，而是按照风险和依赖关系设置。基础架构阶段必须尽早完成，因为后续所有功能都依赖账号、路由、数据库和接口骨架。智能能力阶段时间较长，是因为知识图谱、检索、AI Terminal 和学习规划都需要联调外部服务或复杂数据结构。收尾阶段虽然时间短，但任务密度高，需要提前在前面阶段积累文档和测试材料。

## 四、里程碑清单

- M1：完成课题背景、需求分析和范围裁剪。
- M2：完成基础数据模型与接口骨架。
- M3：完成课程空间与学习现场的主流程。
- M4：完成知识图谱、规划和 AI Terminal 的联动。
- M5：完成课程报告和演示材料。

每个里程碑应对应明确验收标准。M1 的标准是项目名称、用户对象、核心需求和不做事项都能说清楚；M2 的标准是主要数据实体和接口方向稳定；M3 的标准是用户能够进入课程空间并完成学习现场和资料管理主流程；M4 的标准是至少有一条智能能力能够基于资料和学习场景产生结果；M5 的标准是系统、报告、截图和答辩材料口径一致。

里程碑也应绑定具体页面和功能。M3 不仅表示“核心业务完成”，还应能打开 Workspace Shell 的 Workbenches 和 Knowledge 标签页，创建 Workbench 并绑定资料；M4 不仅表示“智能能力完成”，还应能在 AI Terminal 或 AI 助手中基于资料回答问题，能展示 Knowledge Graph 或 Planning 的结果；M5 不仅表示“文档完成”，还应能按答辩脚本连续演示上述流程。

## 五、项目进度网络关系

项目关键依赖关系如下：

```text
需求分析
  -> 架构与数据库建模
  -> 账号/Workspace/Workbench 主流程
  -> 资料库与检索能力
  -> 知识图谱/学习规划/AI Terminal
  -> 学习画像/AI Studio/多模态扩展
  -> 测试回归/课程报告/答辩材料
```

其中，“资料库与检索能力”是后续知识图谱、AI Terminal 和学习规划的共同前置条件；“测试回归/课程报告/答辩材料”依赖全部核心功能完成后进入冻结。

## 六、项目进度表

| 任务包 | 预计开始 | 预计完成 | 关键输出 |
| --- | --- | --- | --- |
| 需求与 WBS | 2026-03-01 | 2026-03-10 | 需求清单、WBS、风险初稿 |
| 架构与建模 | 2026-03-11 | 2026-03-25 | 数据模型、接口草案 |
| 核心功能 | 2026-03-26 | 2026-04-15 | Workspace/Workbench/资料库 |
| 智能功能 | 2026-04-16 | 2026-05-20 | 图谱、规划、AI Terminal |
| 扩展能力 | 2026-05-21 | 2026-06-10 | 学习画像、Studio、多模态 |
| 文档收尾 | 2026-06-11 | 2026-06-17 | 11 份课程报告、PPT、说明 |

进度表中的“文档收尾”不是最后才开始写文档，而是在最终阶段集中校对、补充和统一格式。前期应随开发同步记录需求变更、风险变化、测试问题和成本口径，否则最后阶段会出现大量回忆式补写，降低报告可信度。

## 七、项目进度基准

本项目以 2026-06-17 作为课程提交基准日期。所有后续变更均按以下原则处理：

- 不影响核心演示流程的增强项，可延期。
- 影响报告一致性的内容，必须在提交前修正。
- 影响系统启动和主流程演示的问题，优先级最高。
- 与 3 人团队资源冲突的任务，优先按课程交付价值排序。
- 依赖外部模型、云数据库或部署环境的任务，必须预留备用方案。

进度基准一旦确定，就应作为范围控制和风险控制的依据。若新增功能无法在基准日期前完成测试和文档同步，则不应进入必交范围。对于已经完成但不稳定的功能，可以作为展示增强项处理，不强行纳入核心演示路径。

## 八、项目日历

- 每周进行一次进度回顾。
- 每个阶段结束前做一次里程碑检查。
- 课程提交前一周进入冻结期，原则上不再新增大功能。
- 6 月 17 日前完成最终提交材料整理。

每周进度回顾建议固定检查四类问题：本周完成了什么、下周最重要的交付物是什么、当前最大的阻塞是什么、哪些文档需要同步修改。对于 3 人团队而言，会议不需要过长，但必须把任务状态说清楚，避免成员之间对完成度判断不一致。

## 九、进度控制计划

进度控制重点关注：

1. 哪些工作是否按时完成。
2. 哪些模块存在依赖阻塞。
3. 评审和修改是否挤占最终提交时间。
4. 是否出现范围失控或重复开发。

控制方法包括：

- 每周检查任务状态。
- 记录阻塞项和处理人。
- 以里程碑为准，不以“看起来差不多”为准。
- 发现延期时优先砍掉非核心增强项。
- 使用核心演示链路作为进度判断标准，而不是只看功能数量。
- 对 Codex 辅助生成的内容安排复核时间，避免后期集中返工。
- 对阿里云数据库和部署环境设置检查点，提前发现配置问题。

进度控制需要和风险管理联动。如果智能服务不稳定，就要及时切换到缓存数据或静态演示；如果文档来不及完整扩写，就要优先保证 11 份报告的结构、术语和关键数据一致；如果全栈开发任务过重，就要减少低优先级增强项，把资源集中到核心闭环。

对于本项目，后期新增功能必须通过“是否增强核心学习闭环”来判断优先级。例如，修复 Workbench 状态保存、资料删除后面板同步关闭、AI Terminal 历史会话恢复、知识索引失败提示等问题，优先级高于新增一个没有测试的复杂视觉效果。因为前者直接影响用户连续学习体验，后者更多属于展示增强。

## 十、项目状态报告计划

建议采用“周报 + 阶段复盘”的方式。

- 周报内容：已完成事项、下周计划、阻塞问题、风险变化。
- 阶段复盘：里程碑完成情况、需求变更、测试缺陷和文档一致性。

状态报告不需要追求复杂格式，但要能反映真实进展。建议每次记录包含任务名称、负责人、当前状态、预计完成时间和需要支持的事项。对于课程报告相关工作，还应记录哪些报告已经完成初稿，哪些报告需要根据新功能或新成本口径修改。

## 十一、进度延误处理措施

- 将延期任务拆小，优先完成可提交部分。
- 对低优先级功能降级处理。
- 文档与演示同步准备，避免最后一天集中爆炸。
- 必要时冻结新增需求，确保提交节点。
- 对依赖外部服务的功能准备本地替代数据。
- 将报告扩写、截图、PPT 和演示脚本并行推进。
- 对无法稳定演示的高级功能，改为在报告中说明设计思路和后续计划。

延误处理的核心原则是保护最终交付。若某个模块已经明显无法按期完成，不应继续消耗大量时间追求完整实现，而应尽快确定可展示版本、降级方案和文档说明。这样可以避免单点延期拖垮整个项目。

## 十二、补充说明

本项目的进度计划不是单纯的日历排期，而是把课程报告、实现与演示放在同一张时间表里。这样才能保证“项目一致性”而不是“代码一套、文档一套”。

进度计划还应体现项目的真实组织特点。3 人团队虽然灵活，但抗风险能力相对有限，因此更需要清晰的里程碑、冻结机制和备用方案。只要按核心闭环推进，把增强功能控制在可承受范围内，项目可以在 2026-06-17 前完成课程提交和演示准备。
