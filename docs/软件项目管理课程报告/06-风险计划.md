# 06 风险计划

## 一、风险计划说明

本报告中的风险计划按教材要求理解为风险对策计划，重点在于提前识别风险、评估风险、制定应对策略、设置触发标志和明确责任人。

本项目的风险具有课程项目和智能系统项目的双重特点。一方面，课程提交时间固定，报告、PPT、截图和演示都必须在规定节点前完成；另一方面，项目包含大模型工具、知识图谱、检索、多模态分析和云数据库等依赖，技术不确定性高于普通管理系统。因此，风险计划需要提前设置降级方案，而不是等问题发生后再临时补救。

风险计划的目标不是完全消除风险，而是让团队在风险出现时能够快速判断影响范围，并采取可执行措施。对于 3 人团队来说，风险责任必须明确到角色，避免出现“大家都知道有问题，但没人负责处理”的情况。

## 二、风险识别

本项目的主要风险来自需求、技术、进度、协作、演示和质量六个方面。

需求风险主要表现为智能学习平台功能边界较大，容易把所有想法都加入必做范围；技术风险主要来自外部模型服务、Codex 辅助生成内容、阿里云数据库连接和多模块联调；进度风险主要来自课程提交节点固定、后期文档工作密集；协作风险来自团队人数少、角色兼任较多；演示风险来自环境配置、网络、端口和数据准备；质量风险来自测试覆盖不足、文档与实现不一致和高级功能不稳定。

这些风险都可以落到具体业务模块上。Workspace 是数据边界，一旦所有权校验或删除复制逻辑出错，会影响用户资料安全；Workbench 是学习任务现场，一旦状态保存、资源绑定或多面板恢复出错，会影响连续学习体验；资料库和知识索引是 AI 问答、图谱和规划的基础，一旦抽取、切块或检索失败，智能能力就会失去依据；AI Terminal 和 Agentic 模式涉及写入操作，若审批和降级处理不清晰，可能造成误创建文件、误生成计划或演示中断。

## 三、风险评估与应对策略

| 风险 | 触发标志 | 影响 | 应对策略 | 责任人 |
| --- | --- | --- | --- | --- |
| 需求范围扩张 | 新功能不断被加入 | 进度与文档失控 | 冻结范围，优先核心功能 | 项目负责人 |
| 智能服务不稳定 | 接口超时、返回异常 | 影响演示与联调 | 准备降级方案和缓存结果 | 智能/数据开发 |
| 数据模型反复调整 | 表结构频繁变动 | 返工成本高 | 先定核心实体，再扩展 | 后端开发 |
| 进度延期 | 关键里程碑未完成 | 影响收尾 | 拆分任务，压缩非核心项 | 全体 |
| 协作沟通不畅 | 任务重复或遗漏 | 返工、冲突 | 每周同步，明确负责人 | 项目负责人 |
| 测试覆盖不足 | 缺陷集中暴露 | 演示失败 | 提前回归，建立冒烟测试 | 测试与文档 |
| 安全与权限问题 | 跨用户数据访问 | 风险较高 | 统一认证和数据隔离 | 后端开发 |
| 文档不一致 | 代码与报告口径不同 | 影响评分 | 文档与实现同步更新 | 文档负责人 |
| 环境配置失败 | 本地无法启动 | 演示受阻 | 保留最小可运行环境 | 后端开发 |
| 临近提交压力 | 大量事项堆积 | 质量下降 | 提前冻结并预留缓冲 | 项目负责人 |
| Codex 生成内容错误 | 代码或文档未复核直接使用 | 功能异常或报告失真 | 人工审查、运行验证、保留修改记录 | 全栈开发 |
| 云数据库异常 | 阿里云数据库连接失败或配置错误 | 数据不可用、演示中断 | 备份连接配置，准备本地备用数据 | 运维测试 |
| 关键人员时间不足 | 全栈开发或测试任务无法按期完成 | 多模块延期 | 降低增强项优先级，集中完成主流程 | 项目负责人 |
| Workbench 状态丢失 | 刷新或重新进入后布局、打开资源、AI Studio 结果丢失 | 学习现场不可持续 | 检查 stateJson 保存与恢复，准备固定演示现场 | 全栈开发 |
| 资料索引失败 | PDF、网页、音视频或文本无法抽取/切块 | RAG、图谱和规划依据不足 | 提前索引演示资料，保留已处理样例 | 运维测试 |
| Agent 写操作误触发 | 未经确认创建文件、计划或 Workbench | 用户数据被意外修改 | 写工具先 proposal，审批后执行 | 全栈开发 |
| 学习画像依据不足 | 学习事件、记忆或掌握度信号缺失 | 个性化建议空泛 | 用演示数据补足学习事件和记忆样例 | 需求与研究 |

风险评估时应同时考虑发生概率和影响程度。对于高概率高影响风险，例如需求范围扩张和时间压缩，应在计划阶段就采取预防措施；对于低概率高影响风险，例如数据库连接失败和权限安全问题，应准备应急方案；对于中等风险，例如性能问题和文档遗漏，应通过阶段检查持续监控。

## 四、风险防范措施和处理措施

| 风险类型 | 防范措施 | 发生后的处理措施 |
| --- | --- | --- |
| 需求风险 | 需求冻结、变更审批、范围基线 | 延期低优先级功能，优先保证核心流程 |
| 技术风险 | 提前验证关键接口，保留降级路径 | 使用缓存数据、静态演示或回退实现 |
| 进度风险 | 每周检查里程碑，预留缓冲 | 压缩增强项，集中人力处理阻塞 |
| 质量风险 | 评审、冒烟测试、文档校对 | 建立缺陷清单，按严重级别修复 |
| 环境风险 | 准备本地最小运行环境 | 启用备用端口、备用数据和备用演示 |
| 成本风险 | 控制 Codex、模型和云服务额度 | 暂停非必要调用，使用已有样例数据 |
| 人员风险 | 明确责任边界和阶段交付物 | 调整任务优先级，减少低价值功能 |

防范措施强调提前建设能力，处理措施强调风险发生后的快速止损。例如，智能服务不稳定时，防范措施是提前验证接口和准备缓存结果；发生后则应立即切换到缓存、静态演示或文字说明。云数据库异常时，防范措施是保留连接配置和备份数据；发生后则应切换本地环境或备用截图，确保答辩流程不中断。

## 五、风险触发后的处理原则

1. 先保证可交付。
2. 先处理高概率高影响风险。
3. 先保核心流程，再保扩展能力。
4. 不为单个风险牺牲整个提交节点。
5. 先恢复主流程，再分析深层原因。
6. 风险处理结果要同步到报告、测试记录和演示脚本。

风险触发后，团队应避免陷入长时间排查而忽略提交节点。若问题短时间内无法彻底解决，应先完成可运行版本或降级版本，再安排后续修复。风险处理结束后，需要记录触发原因、处理过程和剩余影响，为风险管理报告和项目总结提供依据。

## 六、风险责任分配

- 项目统筹：范围、进度、成本和文档一致性风险。
- 全栈开发负责人：前端、后端、数据库模型、接口、权限和智能能力接入风险。
- 运维测试负责人：部署环境、阿里云数据库、演示数据、测试回归和备用方案风险。
- 需求与研究负责人：需求边界、开源项目参考、论文方向、课程报告口径和功能优先级风险。

虽然团队规模较小，但风险责任不能模糊。每个风险都应至少有一个主责人和一个协助人，主责人负责判断风险状态和提出处理方案，协助人负责提供验证、资料或环境支持。

## 七、风险监控办法

- 每周更新风险清单。
- 对高风险项设置明确触发阈值。
- 风险变化要同步到进度计划和质量计划。
- 一旦风险落地，立刻启动应急措施，而不是继续观察拖延。
- 在每个里程碑检查时重新评估风险概率和影响。
- 对高风险项保留处理记录，便于后续总结。
- 将风险变化同步到成本计划和进度计划，例如外部服务消耗增加或收尾时间被压缩。

风险监控可以采用简单表格记录，不必建立复杂系统。表格字段建议包括风险编号、风险名称、当前等级、触发标志、责任人、处理状态、下一步动作和关闭条件。只要记录持续更新，就能避免风险长期停留在口头层面。

## 八、应急响应

若风险已发生，优先采取以下动作：

1. 回退到可运行版本。
2. 降级非核心功能。
3. 用可演示材料替代复杂功能。
4. 保证课程提交节点和答辩材料完整。
5. 通知相关成员同步修改测试记录、截图和报告表述。
6. 对外部服务类问题保留错误现象和替代方案说明。

应急响应还需要考虑演示场景。答辩现场或课程检查时，最重要的是能够稳定展示项目价值。因此，团队应提前准备一条最短演示路径，包括登录、进入课程空间、查看资料、调用智能能力或展示生成结果。即使高级功能不可用，也能通过这条路径说明项目核心设计。

针对业务模块的应急策略也要更具体：知识图谱实时生成失败时，展示已生成的课程概念和先修关系；AI Terminal 模型不可用时，展示历史会话和已保存的生成计划；AI Studio 渲染失败时，展示生成的 Markdown、测验或闪卡文本产物；音视频分析不可用时，展示文本资料分析链路；阿里云数据库异常时，切换到本地演示 Workspace。这样每个风险都有对应的产品级降级路径。

## 九、补充说明

风险计划的关键不在于“列很多风险”，而在于每个风险都能落到责任人、触发标志和动作上。这样后续的风险管理报告才能基于真实执行情况继续展开。

本项目风险计划应与质量计划、进度计划和成本计划共同使用。进度计划决定风险发生时哪些任务可以延期，质量计划决定风险处理后的验收标准，成本计划决定可用于模型调用、云服务和演示材料的预算边界。四类计划相互对应，才能形成完整的软件项目管理闭环。
