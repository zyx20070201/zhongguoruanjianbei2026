# 05 成本计划

## 一、成本构成

软件项目成本一般可分为人力资源成本、资产类成本、管理费用和项目特别费用，也可按直接成本与间接成本划分。本项目采用课程作业口径，重点估算开发与交付成本。

由于本项目同时具有课程作业、软件杯参赛作品和实际原型系统三种属性，成本计划需要区分“课程估算成本”和“真实现金支出”。课程估算成本用于体现软件项目管理中的人力、资源、进度和风险投入，便于与进度计划、质量计划、风险计划形成对应关系；真实现金支出则主要包括 Codex 类大模型辅助开发产品、模型调用、服务器或服务机部署、云数据库和少量演示材料费用。

从实际执行看，团队成员以学习、竞赛和课程交付为主要目标，没有按照企业项目方式支付薪酬，因此人力成本属于折算成本。现金成本可通过控制模型额度、复用个人开发设备、使用轻量云资源和选择合适的数据库规格来压缩，整体可以控制在 1500 元以内。报告中的较高预算数字主要用于课程管理分析，不代表项目真实已经支出的金额。

## 二、成本估算

### 1. 人力成本

| 角色 | 估算工时 | 单价（元/小时） | 小计（元） |
| --- | --- | --- | --- |
| 全栈开发 | 280 | 60 | 16,800 |
| 运维测试与演示 | 110 | 45 | 4,950 |
| 需求分析与研究指导 | 100 | 55 | 5,500 |
| 文档统筹与课程报告整理 | 70 | 50 | 3,500 |
| **合计** | 560 |  | **32,800** |

人力成本的估算依据是项目工作分解结构和实际 3 人分工。全栈开发承担前端、后端、数据库和主要智能能力接入工作，投入占比最高；运维测试与演示负责环境、部署、回归检查和演示材料，主要集中在中后期；需求分析与研究指导负责需求拆解、开源项目调研和论文资料支持；文档统筹工作贯穿全程，尤其在提交前集中投入。

该估算采用课程管理中的机会成本口径，即把成员投入时间折算成项目成本，用于衡量项目规模和资源占用。实际团队内部不发生等额工资支付，因此该部分不计入现金支出。

### 2. 资产类与服务成本

| 项目 | 金额（元） | 说明 |
| --- | --- | --- |
| 开发设备折算 | 3,000 | 个人设备折旧或共享成本，课程估算口径 |
| 数据库/工具/环境 | 1,200 | 本地开发、阿里云数据库和调试环境 |
| Codex 类 AI 辅助开发工具 | 1,000 | 大模型驱动的开发辅助产品或额度 |
| 外部服务与模型调用 | 1,000 | 视实际调用频率调整 |
| 测试与演示材料 | 800 | 截图、PPT、打印等 |
| **合计** | **7,000** |  |

资产类与服务成本中，开发设备折算属于管理估算，并非新增采购。数据库和工具环境主要用于支撑本地开发、云端部署和阿里云数据库连接。Codex 类 AI 辅助开发工具是本项目的重要投入项，它提高了代码生成、问题定位、文档扩写和方案整理效率，但也需要通过人工审核、测试和版本控制来降低生成错误风险。

若只统计真实现金支出，则设备折算可以剔除，测试材料也可以大幅压缩。实际现金成本主要由 Codex 类产品、模型调用额度、云服务器或服务机部署费用、阿里云数据库费用构成，预计总额不超过 1500 元。

成本与业务功能的关系主要体现在三类消耗上。第一，AI Terminal、AI Studio、知识图谱解释、学习规划和多模态分析会消耗模型调用额度；第二，资料上传、文件预览、知识切块、embedding 和向量检索会消耗计算与存储资源；第三，阿里云数据库和服务机部署用于保障 Workspace、Workbench、对话历史、学习事件、闪卡和生成物等数据在演示中的稳定访问。因此成本控制不能只按技术类别看，还要按业务链路判断投入是否值得。

### 3. 不可预见成本

按总成本的 10% 预留不可预见费用，用于需求变更、重做和临时服务开销。

不可预见费用主要用于三类情况：第一，智能能力接入过程中模型额度消耗超过预期；第二，云端部署或数据库配置需要临时调整规格；第三，临近提交时发现文档、截图、演示材料需要返工。由于本项目规模仍属于课程原型，预留比例不宜过高，10% 能够覆盖常见波动。

| 现金支出项目 | 估算金额（元） | 控制方式 |

## 四、直接成本与间接成本

- 直接成本：开发人力、外部服务、测试演示材料。
- 间接成本：设备折算、沟通协调、文档整理、会议时间。

直接成本中，开发人力和 Codex 类工具对项目产出影响最大，决定了功能完成速度和文档整理效率。外部服务与模型调用虽然金额不高，但对 AI Terminal、知识图谱生成、多模态分析等功能有直接影响，需要在演示前确认可用性。间接成本主要体现在沟通、返工和材料整理上，若需求频繁变化或文档口径不统一，间接成本会快速上升。

因此，成本控制不能只看账面金额，还要关注返工次数、等待时间和问题定位时间。对于 3 人团队来说，减少沟通误差和保持任务边界清晰，本身就是最重要的成本节约措施。

## 五、费用控制

### 1. 控制原则

1. 控制范围，避免功能不断膨胀。
2. 控制返工，保持接口和文档同步。
3. 控制服务消耗，能本地完成的尽量本地完成。
4. 控制演示成本，把预算优先给最能展示成果的模块。
5. 控制 AI 工具依赖，生成内容必须经过人工复核。
6. 控制云资源规格，课程演示阶段以稳定可用为主，不追求过度配置。

### 2. 控制措施

- 每个阶段结束时核对实际投入与预算差异。
- 对高成本功能设置审批阈值。
- 及时复用已有组件和数据结构。
- 对临时新增需求使用不可预见费，不挤占核心预算。
- Codex 类工具主要用于明确任务、生成初稿、辅助重构和文档扩写，关键代码仍要经过运行验证。
- 阿里云数据库和部署环境按演示需求配置，提交前进行连接测试和备份检查。
- 对多模态分析、复杂 Agent 编排等高成本增强功能设置优先级，必要时采用降级演示。

成本控制应与进度控制同步进行。若某项功能同时消耗大量开发时间、模型调用额度和测试资源，但对核心演示帮助有限，应及时降级或延期。相反，若某项投入能显著提升课程展示效果，例如稳定部署环境、统一演示数据、完善关键报告，则应优先保障。

例如，AI Studio 的复杂交互式可视化资源生成虽然展示效果强，但若每次生成都需要较多模型调用、渲染调试和人工修复，就不适合作为高频演示路径；相反，基于资料生成测验、闪卡、学习计划和 Markdown 报告更贴近学习业务，也更容易稳定展示。成本计划应鼓励把预算优先投向这些能证明产品价值的功能。

## 六、成本计划制订步骤

1. 依据 WBS 先估算各模块成本，补齐遗漏项。
2. 根据项目实际环境做综合调整，并预留不可预见费。
3. 将可执行预算写入计划并作为后续控制基准。
4. 区分课程估算成本和真实现金支出，分别用于报告分析和实际执行。
5. 根据风险计划调整预算缓冲，例如为云部署、模型服务和演示材料预留余量。
6. 在项目总结阶段对预算与实际投入进行对比，分析偏差原因。

本项目制订成本计划时，先根据功能模块估算工作量，再结合 3 人团队的实际职责进行角色拆分，最后根据 Codex 类工具、云数据库和部署服务等真实支出补充现金预算。这样既能满足课程报告对成本构成的要求，也能反映项目真实执行特点。

## 七、补充说明

本项目不是企业级商业系统，因此成本计划的重点不在“精确到分”，而在“有清晰口径、有预算边界、有控制动作”。只要课程作业和软件杯项目口径一致，成本计划就能支撑后续的风险、质量和总结文档。

需要特别说明的是，报告中的 43,780 元总预算是把人力时间、设备折算和服务资源全部纳入后的管理估算，用来衡量项目复杂度和资源占用；1500 元以内是团队真实现金支出的控制目标，用来指导实际采购和部署。两个数字服务于不同目的，并不矛盾。

后续若项目进入更长期的产品化阶段，成本结构会发生变化，服务器、数据库、模型调用、运维监控和安全合规成本都会提高。当前课程阶段应坚持轻量化原则，把钱花在最能保证交付的地方，即 AI 辅助开发工具、稳定部署环境和必要的模型服务上。
