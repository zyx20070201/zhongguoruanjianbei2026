# 03 资源计划

## 一、资源分类

根据教材要求，软件项目资源主要包括人力、可复用的软件构件或组件，以及软硬件环境。本项目对应资源如下。

资源计划的目的不是简单列出团队拥有哪些资源，而是说明这些资源如何服务于项目目标、在不同阶段如何投入、出现冲突时如何调配。本项目的交付物包括可运行系统、课程报告、演示材料和软件杯相关文档，因此资源计划既要覆盖开发资源，也要覆盖测试、部署、研究和文档资源。

### 1. 人力资源

- 全栈开发负责人
- 运维测试与演示负责人
- 需求分析与研究指导负责人
- 项目统筹与课程报告协作人员

本项目实际团队规模为 3 人。由于人数较少，角色之间存在一定兼任关系，但每个成员的主要责任边界仍需清晰。全栈开发负责人承担主要工程实现，运维测试与演示负责人保障环境稳定和质量检查，需求分析与研究指导负责人负责需求口径、开源项目调研和论文方向参考。课程报告由全体协作完成，但需要统一术语和时间线。

### 2. 软件资源

- 代码仓库与版本控制工具
- React、Vite、TypeScript、Node.js、Express、Prisma
- 数据库与迁移工具
- 阿里云数据库及连接配置
- Codex 类 AI 辅助开发工具
- 课程报告模板与图表素材
- 可复用的页面组件、接口封装、数据模型

软件资源中，Codex 类工具主要用于辅助代码实现、问题定位、文档扩写和方案梳理，但其输出必须经过人工复核。阿里云数据库用于提升部署和演示阶段的数据稳定性，本地数据库环境则用于快速开发和调试。两类环境需要保持数据模型和配置说明一致，避免本地可运行但演示环境失败。

从项目业务看，可复用软件资源不仅是框架和工具，还包括已经形成的业务服务和组件。例如 Workspace Shell、Workbench 多面板编辑器、文件系统服务、知识索引服务、课程知识图谱服务、AI Studio 产物生成服务、对话历史服务和学习规划相关接口，都是后续功能迭代和演示组织可以复用的资产。资源计划应把这些业务资产纳入管理，避免重复实现相同的资料读取、上下文构建和生成物保存逻辑。

### 3. 硬件与环境资源

- 开发笔记本电脑
- 本地数据库环境
- 浏览器与调试工具
- 可能的云服务或模型服务访问权限
- 服务机或轻量云服务器
- 网络环境、演示设备和备用截图材料

硬件资源以个人开发设备为主，不额外采购专用服务器。演示阶段需要重点保障网络、浏览器、数据库连接和模型服务可用性。若外部服务临时不可用，应提前准备本地备用数据、静态截图和降级演示路径。

## 二、资源计划

### 1. 人力资源计划

| 角色 | 主要职责 | 投入方式 |
| --- | --- | --- |
| 全栈开发 | 页面、路由、API、认证、数据模型、智能能力接入 | 全程高强度投入 |
| 运维测试 | 部署、数据库连接、测试用例、回归、截图、演示脚本 | 中后期重点投入 |
| 需求与研究 | 需求分析、开源项目调研、论文资料、技术路线建议 | 前期和关键评审阶段投入 |
| 文档协作 | 课程报告、答辩材料、口径校对和总结复盘 | 全程滚动投入 |

人力资源投入具有阶段差异。前期需求与研究投入较多，用于明确赛题要求、课程要求和项目范围；中期全栈开发投入最高，用于完成核心功能和接口联调；后期运维测试与文档协作投入增加，用于保证演示稳定、报告一致和提交材料完整。

### 2. 软硬件资源计划

- 开发环境：统一使用本地开发机与浏览器环境。
- 数据环境：早期使用 SQLite 快速开发，后续迁移到阿里云数据库支撑演示和部署。
- 服务环境：按需接入检索、向量、模型或外部接口。
- 产物环境：课程报告、截图、演示稿、答辩材料统一归档。

软硬件资源使用应遵循“开发轻量、演示稳定、资料可追溯”的原则。本地环境用于快速验证功能，云端环境用于正式演示和远程访问。所有关键配置，如数据库连接、环境变量、模型服务密钥和演示账号，都应有记录并妥善保护，避免因个人设备或临时环境变化影响交付。

演示资源还应按业务场景准备，而不是只准备空系统。建议至少准备一个“数据库系统复习”或“大模型系统课程学习”的 Workspace，包含课件/PDF、网页资料、代码片段、一个已创建的 Workbench、若干知识图谱概念、一个学习计划、几条 AI Terminal 对话、AI Studio 生成的测验或闪卡。这样资源投入可以直接服务于答辩展示和质量检查。

## 三、项目组织结构

建议采用“3 人核心团队 + 明确职责边界”的轻量组织。

```text
项目统筹（共同承担）
├─ 全栈开发负责人
├─ 运维测试与演示负责人
└─ 需求分析与研究指导负责人
```

该组织结构的优势是沟通路径短、决策速度快，适合课程项目和竞赛原型；不足是关键工作容易集中在少数成员身上，因此必须通过任务清单、阶段检查和文档记录降低单点风险。每个阶段结束时，团队需要确认功能完成情况、环境状态、报告同步情况和下一阶段资源重点。

组织结构也对应系统模块责任。全栈开发负责人重点关注 Workspace/Workbench 主流程、资料库、AI Terminal、AI Studio 和后端服务集成；运维测试负责人重点关注登录、数据隔离、阿里云数据库、上传下载、知识索引、演示账号和浏览器运行状态；需求与研究负责人重点关注赛题中的“个性化资源生成”“学习多智能体系统”“知识图谱与学习规划”等要求是否被真实功能覆盖。

## 四、责任分配矩阵

| 任务 | 负责人 | 参与人 | 说明 |
| --- | --- | --- | --- |
| 需求与范围控制 | 需求分析与研究负责人 | 全体 | 保证课程、赛题和实现一致 |
| 前端功能 | 全栈开发负责人 | 运维测试 | 页面与演示流程联调 |
| 后端服务 | 全栈开发负责人 | 运维测试 | 接口、认证、数据流 |
| 智能能力 | 全栈开发负责人 | 需求分析与研究负责人 | 检索、图谱、规划、Agent |
| 部署与数据库 | 运维测试负责人 | 全栈开发负责人 | 阿里云数据库、服务机、环境变量 |
| 测试与回归 | 运维测试负责人 | 全体 | 修复缺陷并补记录 |
| 课程报告 | 需求分析与研究负责人 | 全体 | 统一口径和版本 |

责任分配矩阵需要随项目推进动态校正。若某项任务出现延期，应先判断是人力不足、依赖阻塞还是需求不清，再决定是否调整负责人或降低功能优先级。

## 五、资源计划编制依据

- 项目范围说明书：当前项目功能说明和需求分析。
- 项目工作分解结构：按 Workspace、Workbench、资料库、知识图谱、AI Terminal、规划、画像、Studio、测试与文档拆分。
- 历史项目数据：参考当前仓库已有模块、设计文档和既有实现。
- 管理政策：优先保障课程节点与答辩展示，控制范围蔓延。
- 活动工期估算：结合现有完成度分配资源。
- 其他制约因素：人手、时间、接口稳定性、演示环境。
- 成本计划：现金支出控制在 1500 元以内，课程估算成本按完整资源投入计算。
- 风险计划：对智能服务、云数据库、演示环境和关键人员投入设置备用方案。

## 六、资源使用原则

1. 核心模块优先，先完成能形成闭环的功能。
2. 共享资源优先，避免重复实现相同页面或逻辑。
3. 资料与代码同步归档，减少后期补文档成本。
4. 后期集中保障测试、截图、演示和报告排版资源。
5. AI 辅助工具优先用于提高效率，但不能替代人工确认和测试。
6. 云端资源按需使用，避免为非核心功能消耗过多费用。
7. 关键配置和资料统一归档，减少因个人设备差异造成的重复工作。

## 七、补充说明

资源计划不仅是“有多少人和机器”，还要明确“谁负责什么、什么时候用什么资源、资源如何切换”。对于本项目来说，最关键的资源是稳定的开发节奏、统一的数据模型和持续的文档一致性。

由于团队只有 3 人，资源计划还承担着降低协作风险的作用。每名成员都需要知道自己的主要交付物，也需要了解其他成员的依赖关系。例如，需求分析结果会影响全栈开发的模块划分，数据库和部署配置会影响运维测试的演示准备，质量检查结果又会反向影响报告修改和功能收尾。

总体来看，本项目资源数量不多，但资源类型较完整。只要能够合理使用 Codex 类工具、云数据库、本地开发环境和现有设计文档，就可以在有限时间内完成课程报告和演示原型的交付。

资源计划最终要服务于真实业务闭环：资料要能进入 Workspace，资源要能绑定到 Workbench，知识块要能支撑检索和问答，学习事件要能支撑画像和计划，生成物要能保存并再次作为资料使用。后续资源调度应优先保障这些链路，而不是平均分配到所有设想功能。
